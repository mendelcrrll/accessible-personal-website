Accessibility Techniques

Deep dive into specific accessibility implementations and best practices

Implementation Techniques

Semantic HTML

Using proper HTML elements provides structure and meaning to content.

Key Elements:

  • <header> - Page header and site navigation
  • <nav> - Navigation links (can be multiple)
  • <main> - Main content area (only one per page)
  • <article> - Self-contained content
  • <section> - Grouped related content
  • <aside> - Sidebar or supplementary content
  • <footer> - Footer with contentinfo role

Example:

<header role="banner">
    <nav role="navigation" aria-label="Main navigation">
        <!-- Navigation items -->
    </nav>
</header>

<main id="main-content">
    <section>
        <h2>Section Title</h2>
        <!-- Content -->
    </section>
</main>

ARIA Labels and Descriptions

ARIA attributes provide additional context to assistive technologies.

Common ARIA Attributes:

  • aria-label - Provides an accessible name for an element
  • aria-labelledby - Links to an element that labels this one
  • aria-describedby - Provides additional description
  • aria-expanded - Indicates if expandable content is open
  • aria-hidden - Hides element from assistive technologies
  • aria-live - Announces dynamic content changes
  • aria-current - Marks the current page/section

Example:

<button 
    aria-label="Close menu"
    aria-expanded="false"
    aria-controls="menu"

<input 
    type="email"
    aria-describedby="email-help"
>

Keyboard Navigation

Ensure all functionality is accessible via keyboard without mouse.

Best Practices:

  • All interactive elements must be keyboard accessible
  • Tab order should follow logical flow (use tabindex sparingly)
  • Provide visible focus indicators
  • Implement keyboard shortcuts for common actions
  • Use Escape to close modals and menus
  • Arrow keys for navigation within components
  • Enter/Space to activate buttons and links

Example Focus Styling:

button:focus {
    outline: 2px solid #ffc107;
    outline-offset: 2px;
}

/* Support for keyboard-nav only focus */
button:focus-visible {
    outline: 2px solid #ffc107;
}

Color Contrast

Sufficient color contrast ensures text is readable by people with low vision.

WCAG Requirements:

  • Normal text (AA): 4.5:1 contrast ratio
  • Large text (AA): 3:1 contrast ratio
  • UI components (AA): 3:1 contrast ratio
  • Graphical objects (AA): 3:1 contrast ratio

Testing Tools:

  • WebAIM Contrast Checker
  • Contrast Ratio Tool
  • Chrome DevTools
  • Color Blindness Simulator

Example:

/* Good contrast */
body {
    color: #212529;      /* Dark gray */
    background: #ffffff; /* White */
    /* Ratio: 8.6:1 */
}

/* Avoid */
color: #999999;      /* Gray */
background: #f0f0f0; /* Light gray */
/* Ratio: 3.15:1 - May fail for normal text */

Form Accessibility

Accessible forms are essential for user interaction.

Guidelines:

  • Associate labels with form inputs using for and id
  • Use aria-label or aria-labelledby if label element not suitable
  • Provide aria-describedby for help text
  • Clearly indicate required fields
  • Use aria-invalid for validation errors
  • Provide clear error messages
  • Use fieldset and legend for grouped inputs
  • Support autocomplete attributes

Example:

<div class="form-group">
    <label for="email">
        Email <span aria-label="required">*</span>
    </label>
    <input 
        type="email"
        id="email"
        required
        aria-describedby="email-help"
        aria-invalid="false"
    >
    <small id="email-help">We'll never share your email</small>
</div>

Focus Management

Proper focus management improves navigation and usability.

Techniques:

  • Maintain logical focus order with proper HTML structure
  • Use skip links for keyboard users
  • Implement focus trapping in modals
  • Restore focus when closing modals
  • Use autofocus sparingly
  • Provide visible focus indicators
  • Announce focus changes to screen readers

Skip Link Example:

<a href="#main-content" class="skip-link">
    Skip to main content
</a>

/* CSS */
.skip-link {
    position: absolute;
    top: -40px;
    left: 0;
}

.skip-link:focus {
    top: 0;
}

Screen Reader Support

Ensure content is properly announced to screen reader users.

Techniques:

  • Use proper heading hierarchy
  • Include alternative text for images
  • Use aria-live regions for dynamic content
  • Provide descriptive link text (avoid "click here")
  • Use .sr-only class for visual-only content
  • Announce form errors with aria-alert
  • Test with actual screen readers (NVDA, JAWS, VoiceOver)

Screen Reader Only Text:

.sr-only {
    position: absolute;
    width: 1px;
    height: 1px;
    padding: 0;
    margin: -1px;
    overflow: hidden;
    clip: rect(0, 0, 0, 0);
    white-space: nowrap;
    border-width: 0;
}

<p class="sr-only">This is only read by screen readers</p>

Responsive and Accessible Design

Responsive design benefits all users, especially those with zoomed content.

Guidelines:

  • Support zoom up to 200% without loss of functionality
  • Use flexible layouts (flexbox, grid)
  • Test touch targets are at least 44x44px (CSS pixels)
  • Ensure mobile navigation is keyboard accessible
  • Test on multiple devices and screen sizes
  • Avoid small, hard-to-tap targets
  • Provide alternatives to hover-based interactions

Example:

/* Accessible touch targets */
button {
    min-width: 44px;
    min-height: 44px;
    padding: 0.75rem 1rem;
}

/* Flexible layout */
@media (max-width: 768px) {
    .features-grid {
        grid-template-columns: 1fr;
    }
}

WCAG 2.1 Compliance Levels

Understanding WCAG compliance levels helps prioritize accessibility efforts.

Level A (Basic):

  • Alternative text for images
  • Proper color contrast (3:1)
  • Keyboard accessible
  • Page title

Level AA (Recommended):

  • Enhanced color contrast (4.5:1)
  • Proper heading structure
  • Form labels
  • Focus visible
  • Meaningful link text

Level AAA (Advanced):

  • Enhanced color contrast (7:1)
  • Sign language for audio content
  • Extended audio descriptions
  • Readability (lower secondary school level)

Testing and Tools

Multiple tools and methods help ensure accessibility compliance.

Automated Testing:

  • axe DevTools (browser extension)
  • Lighthouse (Chrome DevTools)
  • WAVE (browser extension)
  • WebAIM Tools
  • Pa11y (command-line tool)

Manual Testing:

  • Keyboard-only navigation (no mouse)
  • Screen reader testing (NVDA, JAWS, VoiceOver)
  • Color contrast checking
  • Zoom testing (200%)
  • Browser zoom and text resize

User Testing:

  • Test with actual users with disabilities
  • Gather feedback on usability
  • Identify pain points and barriers
